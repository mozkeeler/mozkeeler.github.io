Leaky Input Parsing in nsConvertToActualKeyGenParams

Apr 16th, 2014

Consider the following code (modified for clarity):

The purpose of nsConvertToActualKeyGenParams is to parse and process a string consisting of semicolon-delimited key:value pairs. The known keys are "curve" and "popcert". The function getNextNameValueFromECKeygenParamString tokenizes the input. The only state that function keeps is how far along in params it has gotten. Let’s assume there’s nothing wrong with it. If you’re the suspicious sort, you may have already noticed what can go wrong with nsConvertToActualKeyGenParams. What happens if a string like "curve=secp521r1;curve=secp521r1" is given as its input? curve will first point to a freshly-allocated copy of "secp521r1" and then it will point to another freshly-allocated copy of "secp521r1". Of course, the issue isn’t that it’s inefficient – it’s that the memory for that first copy has been leaked. It’s even worse if multiple "popcert" values are given, because each copy leaks an entire CERTCertificate.

If this were the argument-parsing code of a locally-run, non-setuid, short-lived utility program, it would probably be no more than a papercut-correctness bug. Sure, some memory gets leaked, but nobody would call it with nonsensical input, right? Unfortunately this is more serious than a local correctness issue. The input to nsConvertToActualKeyGenParams gets passed unchanged from window.crypto.generateCRMFRequest, meaning that any page could trigger this bug.

Looking at the history of this code, it appears to have landed as part of a large commit with no tests. The reviewer even mentions that the key:value parsing should be in its own function, which maybe would have made this bug more obvious. In any case, going forward, it’s clear that large pieces of new code should be broken into smaller pices, tested thoroughly, and reviewed carefully. Additionally, old code with insufficient testing coverage should either be tested or removed if it is no longer necessary.
